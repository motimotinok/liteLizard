# LiteLizard 仕様 v003

最終更新: 2026-02-28

本ドキュメントは v002 を置き換える優先仕様とする。

---

## 1. 概要・方針

- Electron 製のデスクトップ文書エディターアプリ。
- 執筆（ライト）と AI 分析（リザード）を1画面で統合する。
- 文書本体は LiteLizard 専用フォーマット (`.lzl`)、解析結果は JSON で別ファイル保存する。
- 解析はアプリから自社サーバー API にのみ送信する。
- 解析機能はログイン必須。クライアント側にユーザー個別 API キー設定画面は持たない。

---

## 2. ファイル構成

| ファイル / ディレクトリ | 役割 |
|------------------------|------|
| `*.lzl` | 文書本体（LiteLizard 専用フォーマット） |
| `.litelizard/` | 解析結果格納ディレクトリ（隠しフォルダ、ユーザーに非表示） |
| `.litelizard/{documentId}.analysis.json` | 解析結果（段落・章ごと） |

- `.lzl` ファイル内に `documentId`（例: `d_[alphanumeric]{6,32}`）を埋め込む。
- 解析 JSON はプロジェクトルートの `.litelizard/` ディレクトリに `{documentId}.analysis.json` として保存する。
- `.litelizard/` はプロジェクトルートに1つのみ存在し、配下フォルダの `.lzl` ファイルの解析 JSON もすべてここに格納する。
- `.lzl` のファイル名と JSON のファイル名は連動しない。`documentId` が紐付けの正規キーとなる。
- 例: `essay.lzl`（内部に `d_a8f3k2`）↔ `{プロジェクトルート}/.litelizard/d_a8f3k2.analysis.json`
- `documentId` は UI 上には表示しない。
- `.lzl` をアプリ外でリネームしても解析結果との紐付けは維持される。
- `.lzl` を複製した場合、`documentId` の重複を検出し、開いた際に新しい ID を再採番する。

## 3. エディターパネル

VSCode に近い見た目をベースとしつつ、コード記述部分を執筆領域に置き換えた構成。

### 3.1 ミクロ視点（段落編集モード）

- 段落（paragraph）ごとにチャンクとして視覚的に区切って表示する。
- 各チャンクには永続的な `paragraphId` を付与する。
- チャンク単位でドラッグ&ドロップによる並び替えが可能。
- テキスト編集はこの視点で行う。

### 3.2 マクロ視点（構造把握モード）

- 各チャンクを短いカード形式（タイトル・プレビューテキスト等）に縮小して表示する。
- 文書全体の流れ・章構成を俯瞰するための視点。
- テキスト直接編集は行わない（読み取り・構造操作のみ）。


### 3.3 視点切り替え

- `Ctrl + マウスホイール` で切り替える。
- ミクロ↔マクロの **2段階のみ**（中間状態なし）。
- スクロール量が閾値を超えた方向へスナップ切り替え。
- 切り替え時はアニメーション付きで遷移する。

---

### 3.4 縦書き / 横書き切り替え

LiteLizard は視点（ミクロ / マクロ）とは独立して、**縦書き / 横書きの表示モード切り替え**をサポートする。

#### 3.4.1 切り替え仕様

- UI トグルにより「横書き」「縦書き」を切り替え可能とする。
- 視点（ミクロ / マクロ）との組み合わせは以下の4状態を許容する：
  - ミクロ × 横書き
  - ミクロ × 縦書き
  - マクロ × 横書き
  - マクロ × 縦書き

#### 3.4.2 横書きモード

- 一般的なエディターと同様、上から下へスクロールする。
- ページ区切りや用紙境界は表示しない。
- 文書は1つの連続したキャンバスとして扱う。

#### 3.4.3 縦書きモード

- 文字組は縦方向に配置する（top-to-bottom）。
- スクロールは右から左方向へ進行する。
- スクロールに応じて表示領域が左側へ連続的に展開される。

#### 3.4.4 UIコンセプト（巻物型）

- 既存のワープロソフトのような「ページ単位の区切り」は表示しない。
- 紙の切れ目や原稿用紙的な枠線は持たない。
- 縦書き時も文書は **地続きの1枚キャンバス** として扱う。
- 視覚的には「巻物を横に広げていく」ような連続UIとする。

#### 3.4.5 構造との関係

- 縦書き / 横書きの違いは **表示レイヤーの変更のみ** とする。
- paragraphId / chapterId / order 等の内部データ構造には影響しない。
- ドラッグ&ドロップ挙動は表示方向に応じて自然な方向へ適応させる（縦書き時は左右方向を基準とする）。

## 4. 分析ペイン

エディターパネル右側に配置する。エディターの視点と連動して表示内容が切り替わる。

### 4.1 ミクロ視点時

- 段落（paragraph）ごとの分析結果カードを縦に並べて表示する。
- エディター側の段落と上下位置を同期させる（スクロール連動）。
- 各カードに対応する段落の解析結果（モードに応じた出力）を表示する。

### 4.2 マクロ視点時

- 章（chapter）ごとの分析サマリーを表示する。
- サマリーは段落分析の集約ではなく、**章単位で個別に LLM を呼び出して生成する**。
- 章が切り替わるタイミングで対応するサマリーカードをハイライトする。

---

### 4.3 分析カードと段落の相互連動仕様

#### 4.3.1 クリック時の挙動

- 分析ペイン内の段落カードをクリックした場合、
  - 対応する `paragraphId` を持つ段落をエディターパネル側で特定する。
  - 該当段落を一時的にハイライト表示する。

#### 4.3.2 ドラッグ&ドロップ時の挙動

- 分析ペイン側で段落カードをドラッグ&ドロップにより並び替えた場合：
  - 対応する `paragraphId` を基準に、エディターパネル側の段落順序も同様に再計算する。
  - 分析カードの並び順とエディター側の段落順序は常に一致させる。
- 並び替えの正規データは `paragraphId` と `order` に基づき更新される。
- 表示レイヤー（分析ペイン / エディター）は同一データソースを参照する。

## 5. 生成モード

### 5.1 モード一覧

| # | モード名 | 内容 | MVPスコープ |
|---|----------|------|-------------|
| 1 | 読み手の感想予想 | その段落・章を読んだ読者がどう感じるかを予測する | ✅ 優先実装 |
| 2 | 欠点・弱み分析 | 論理の弱点・説明不足・冗長な箇所を指摘する | 将来実装 |
| 3 | 言い換え・発想の拡張 | 別の表現や切り口の提案を行う | 将来実装 |

### 5.2 UI と制約

- ユーザーはモード選択 UI から使用するモードを選択できる。
- **プロンプトの直接編集は不可**。プロンプトは開発者がバックエンドで管理する。
- モードの追加・変更はサーバー側のアップデートで行い、クライアントの更新を要しない設計を目指す。

### 5.3 MVP スコープ

- MVP では「読み手の感想予想」のみを実装する。
- モード選択 UI はスタブとして置くことを許容するが、MVP では選択肢は1つとする。

## 6. 段落データモデル

### 6.1 段落識別子

- 各段落に永続的な `paragraphId` を付与する（例: `p_[alphanumeric]{6,32}`）。
- `paragraphId` はドラッグ&ドロップによる並び替えや本文編集では変わらない。
- 識別子の埋め込み方式の詳細は §13 参照。


### 6.2 章識別子

- 各章に永続的な `chapterId` を付与する（例: `c_[alphanumeric]{6,32}`）。

---

### 6.3 解析ステータス

段落・章それぞれに以下のステータスを持つ。

| ステータス | 意味 |
|------------|------|
| `fresh` | 解析済み、本文との同期が取れている |
| `stale` | 本文が変更され、再解析が必要 |
| `pending` | 解析リクエスト送信中 |
| `failed` | 解析失敗 |

### 6.4 解析 JSON 構造（段落）

```json
{
  "paragraphId": "p_xxxx",
  "chapterId": "c_xxxx",
  "order": 1,
  "status": "fresh",
  "mode": "readerReaction",
  "output": "...",
  "sourceHash": "sha256-of-text",
  "updatedAt": "2026-02-28T00:00:00Z"
}
```

### 6.5 解析 JSON 構造（章サマリー）

```json
{
  "chapterId": "c_xxxx",
  "status": "fresh",
  "order": 1,
  "mode": "readerReaction",
  "output": "...",
  "updatedAt": "2026-02-28T00:00:00Z"
}
```

## 7. 解析実行仕様

### 7.1 段落解析

- 分析ペインの各段落カードから個別に実行する。
- 対象段落を中心に、それが属する**同一章（chapter）内の先頭段落から対象段落まで**を入力コンテキストとして LLM に送信する。
  - 例: 第2章の3番目の段落を解析する場合、第2章の1・2・3番目の段落を入力とする。
- 対象段落より後ろの段落、および他章の段落は入力に含めない。
- 章をまたいだ文脈は段落解析では考慮しない（章サマリー解析で扱う）。

### 7.2 章サマリー解析

- マクロ視点の各章カードから実行する、または全体解析の一部として実行する。
- 章に含まれる全段落テキストを入力として LLM を呼び出す。
- 段落解析とは **独立した** リクエストとして送信する。

### 7.3 全体解析

- 分析ペイン上部のボタンから実行する。
- `stale` / 未解析の段落・章を一括でキューに追加して順次処理する。
- 実行後「成功件数 / 失敗件数」を明示表示する。

---

### 7.4 分析視点の将来的拡張

将来的に、分析視点として以下の2種類を切り替え可能とする設計を想定する。

#### 7.4.1 逐次読者視点（Sequential Reader View）

- 読者が文章を順に読み進めていく過程を模倣する。
- 対象段落までの文脈のみを入力として分析を行う。
- 現在の MVP 実装はこの視点を採用する。

#### 7.4.2 全体俯瞰視点（Global Overview View）

- 文書全体、または章全体を入力として分析を行う。
- 後続段落の内容も含めた上で、対象段落の位置づけや役割を評価する。
- MVP では未実装とし、将来的なモード追加として設計のみ保持する。

---

## 8. 保存と編集確定

- 編集中はインメモリ状態として保持する。
- 本文 (`.lzl`) と解析 (`.json`) の書き込みはユーザーの明示保存時のみ行う。
- ドラッグ&ドロップによる並び替えも、確定は保存時のみ。
- 保存せずにエディターを閉じた場合、ディスク上ファイルは変更しない。

---

## 9. 認証・API

- 未ログイン時は解析 UI を無効化する。
- クライアントは自社サーバー API のみ呼び出す。
- サーバー側で LLM プロバイダーキーを管理する。
- レート制限・トークン使用量の管理はサーバー側で行う。

---

## 10. エラーポリシー

- 段落単位の失敗は `failed` ステータスを保持し、成功した段落の結果は反映する。
- 全体解析時は成功件数 / 失敗件数を明示表示する。
- ネットワーク・認証失敗時は再試行導線を表示する。

## 11. 操作フロー・初期化仕様

本章は、LiteLizard の操作定義・初期化ルール・フォーマット方針を明文化する。

---

### 11.1 新規文書作成フロー

新規文書作成時：

1. `documentId` を1つ生成する。
2. `chapterId` を1つ生成する。
3. `paragraphId` を1つ生成する。
4. 空文字の段落を1つ持つ状態で開始する。
5. 初期状態は「1章・1段落」である。

段落が0の状態は許容しない。

---

### 11.2 既存ファイルインポート

既存テキストを開いた場合の初期化ルール：

#### ID が存在しない場合

- 改行単位で段落を分割する。
- 各段落に `paragraphId` を自動生成する。
- 文書全体を1章とみなし、`chapterId` を1つ生成する。

#### ID が既に存在する場合

- `paragraphId` / `chapterId` を信頼する。
- 重複・欠損チェックを行う。
- 不正または重複があった場合は自動修復する。
- 自動修復時、UI は停止しない。
- 将来的にログ出力は行う（MVP では UI 通知しない）。

---

### 11.3 章（chapter）CRUD

#### 初期状態

- 文書は必ず1章以上を持つ。

#### 章の追加

- ユーザーが明示的に追加する。
- 操作は `Ctrl + Enter`。
- 現在位置で章区切りを挿入する。
- 章は連続した段落集合とする。

#### 章の削除

- 文書に章が1つしか存在しない場合、削除はブロックする。
- 対象章が2章以上存在する場合のみ削除可能。
- **通常の章を削除した場合**：その章の段落は前の章の末尾へ吸収される（`chapterId` は前章の ID に更新される）。
- **最初の章を削除した場合**：その章の段落は次章の先頭にまとめて挿入される。次章の `chapterId` はそのまま維持される。

---

### 11.4 段落（paragraph）生成・削除ルール

#### 生成

- `Enter` キーを押すと現在位置で段落を分割し、新しい段落を生成する。
- 同段落内改行（`Shift+Enter` 等）は実装しない。
- 段落は常に明確に分離された単位とする。

#### 削除・統合

- 段落が**空の状態**で `Delete` / `Backspace` を押すと、その段落を削除する。
- 段落の**先頭**で `Backspace` を押すと、その段落のテキストを前の段落の末尾に統合する（`paragraphId` は前の段落のものに統一）。
- **最後の1段落を削除しようとした場合**：段落は削除されるが章は残る。カーソルは章のタイトル部分へ移動する。その状態で `Enter` を押すと新規段落が作成される。

段落は LiteLizard の最小構造単位であり、段落が0の状態は許容しない。

---

### 11.5 ID 破損時の処理

- ID 欠損・重複・破損が検出された場合、自動修復する。
- UI は停止しない。
- MVP ではユーザー通知は行わない。
- 将来的にログ出力を追加する。

---

### 11.6 Undo / Redo 対象

Undo/Redo の対象に含める：

- テキスト編集
- 段落生成
- 章追加・削除
- ドラッグ&ドロップによる並び替え

構造操作も Undo 対象とする。

---

## 12. エクスプローラーパネル

LiteLizard はアプリ内にファイル・フォルダ管理用のエクスプローラーパネルを持つ。作業単位は「プロジェクト」であり、1つのフォルダをルートとして管理する。

---

### 12.1 プロジェクト管理

#### 新規プロジェクト作成

- アプリ起動時または「新規プロジェクト」メニューから実行する。
- ダイアログでプロジェクト名と作成先ディレクトリを入力・選択する。
- 指定ディレクトリ内にプロジェクト名のフォルダを作成し、`.litelizard/` を初期化する。
- 作成後、エクスプローラーパネルにそのフォルダをルートとして表示する。

#### 既存プロジェクトを開く

- 「プロジェクトを開く」メニューから実行する。
- OS のフォルダ選択ダイアログでプロジェクトのルートフォルダを選択する。
- ルートに `.litelizard/` が存在するフォルダを有効なプロジェクトとして認識する。
- `.litelizard/` が存在しない場合は確認ダイアログを表示し、ユーザーが許可した場合は `.litelizard/` を作成してプロジェクトとして開く。

---

### 12.2 表示仕様

- フォルダ名とファイル名をツリー形式で表示する。
- ファイル名は**拡張子（`.lzl`）を非表示**とする。
- `.litelizard/` ディレクトリおよびその中の JSON ファイルは**表示しない**。
- `documentId` はいかなる UI にも表示しない。

---

### 12.3 操作：アイコンボタン

エクスプローラーパネルのヘッダー部にアイコンボタンを配置する。

| アイコン | 機能 |
|----------|------|
| 新規ファイル | 現在選択中フォルダ内に新規 `.lzl` を作成する |
| 新規フォルダ | 現在選択中フォルダ内に新規フォルダを作成する |

- 作成直後はインライン入力でファイル名 / フォルダ名を入力させる。
- フォルダが未選択の場合はルートディレクトリに作成する。

---

### 12.4 操作：右クリックメニュー

ファイルおよびフォルダを右クリックすると以下のメニューを表示する。

| メニュー項目 | 動作 |
|--------------|------|
| リネーム | インライン入力でファイル名 / フォルダ名を変更する |
| 削除 | 確認ダイアログを表示してから削除する |

---

### 12.5 リネーム仕様

- ファイルのリネームは `.lzl` のファイル名のみ変更する。
- `.litelizard/{documentId}.analysis.json` のファイル名は変更しない（`documentId` ベースのため影響なし）。
- リネーム時に拡張子は自動付与する（ユーザーは拡張子を入力しない）。

---

### 12.6 削除仕様

- `.lzl` を削除する場合、対応する `.litelizard/{documentId}.analysis.json` も合わせて削除する。
- フォルダを削除する場合、配下の `.lzl` に対応するすべての解析 JSON も合わせて削除する。
- 削除前に確認ダイアログを表示する（MVP では「ゴミ箱」機能は持たない）。

---

## 13. .lzl ファイル内部フォーマット

> **本章は未策定です。実装フェーズ開始前に別途作成する予定です。**

### 決定事項

- 内部形式はテキストベースとする。
- `documentId` / `chapterId` / `paragraphId` をメタデータとして埋め込む。
- ユーザーが通常編集する際にメタデータが視覚的に露出しない形式を採用する。
- 他エディターで開いた場合も本文テキストは読み取り可能な状態を維持することが望ましい。

### 未決定事項（本章策定時に決定する）

- メタデータの具体的な埋め込み形式（フロントマター方式・特殊行方式・その他）
- 章区切りの表現方法
- ID 破損時の修復に必要な冗長性の持たせ方
